## 1. プロジェクト概要

### 1.1 背景

既存サイト「占いクラウド」は占い師とユーザーをつなぐ2サイドマーケットプレイスである。現状は占い師 **101名**・鑑定サービス **59件** が登録され、相談形式は電話・チャット・メール・Zoom・対面の5種をサポートする。ユーザー利用は無料、占い師は年額 ¥10,000 の出店モデル。ブログは別ドメイン `contents.uranai.cloud`（WordPress 想定）に分離されている。

### 1.2 目的

| # | 目的 | 内容 |
|---|------|------|
| 目的1 | デザイン水準の刷新 | 参照サイト `brighty.site`（コーチングマッチング）水準のミニマル・上品なUIへ刷新。**オフホワイト基調＋navy（信頼・専門性）＋coral orange（温かみ）の暖色アクセント**（brighty 実測パレット。詳細は `design-system.yml` v1.1.0）、広い余白、カテゴリカード／統計／3ステップCTA／「アドバイザー相談」型マッチング導線を踏襲 |
| 目的2 | ブログの本体統合 | 別ドメインの WordPress ブログを本体に統合し、占い師が **著者** として記事を投稿できるようにする |

### 1.3 技術スタック（確定・変更不可）

| レイヤー | 技術 |
|---------|------|
| フレームワーク | Next.js 15 (App Router) |
| 言語 | TypeScript |
| スタイリング | Tailwind CSS |
| UIコンポーネント | shadcn/ui |
| ORM | Prisma |
| データベース | PostgreSQL |
| 認証・課金 | CC-Auth（共通認証課金基盤） |

## 2. ユーザーロールと権限

| ロール | 説明 | 主な権限 |
|--------|------|---------|
| 一般ユーザー（Seeker） | 鑑定を受けたい相談者 | 占い師・サービス閲覧、検索、マッチング相談、予約リクエスト送信、ブログ閲覧、お気に入り登録 |
| 占い師（Advisor） | 鑑定サービスを提供する出店者 | 自プロフィール・サービス編集、予約リクエスト受信・応答、**ブログ記事投稿（著者）** |
| 運営管理者（Admin） | プラットフォーム運営 | 占い師・ユーザー・サービス管理、ブログ全記事管理・公開承認、カテゴリ・タグ管理、通報対応 |

> **認証基盤**: 3ロールはすべて CC-Auth で認証・ロール管理する。ロールクレームを CC-Auth から受領し、アプリ側で RBAC を実施する。CC-Auth の `.ccagi.yml` 設定（domain / cc_auth セクション）は現時点 **未設定（要設定）** — 「7. 未確定事項・前提条件」参照。

## 3. 機能要件 — MVP

機能要件は以下の3エピックに分類する。各機能はユーザーストーリーと受け入れ条件（Acceptance Criteria, AC）で定義する。

### エピックA: デザイン刷新

> 詳細な美的方向性は `design-requirements.md`、非機能観点は `non-functional.md` を参照。本節は機能的にデザイン刷新が満たすべき構造要件を定義する。

#### A-1. トップページ（ランディング）

**ユーザーストーリー**
> 一般ユーザーとして、トップページを開いたとき、サービスの価値と使い方が一目で理解でき、占い師を探す導線にすぐ到達したい。

**受け入れ条件**
- [ ] AC-A1-1: ヒーローセクションがファーストビューに表示され、キャッチコピーと主要CTA（「占い師を探す」「無料で相談する」）を含む
- [ ] AC-A1-2: 統計セクションを表示する（例: 登録占い師数、累計鑑定数、占術カテゴリ数、平均満足度）。数値はDB集計値または運営設定値を表示し、ハードコードしない
- [ ] AC-A1-3: 占術カテゴリをカード形式のグリッドで表示し、各カードから該当カテゴリの占い師一覧へ遷移できる
- [ ] AC-A1-4: 「3ステップで始める」セクション（例: 登録 → 相談・マッチング → 予約）を表示する
- [ ] AC-A1-5: ピックアップ占い師セクションを表示し、各占い師プロフィールへ遷移できる
- [ ] AC-A1-6: モバイル幅（375px）でレイアウト崩れがなく、CTAがタップ可能（タッチターゲット 44px 以上）

#### A-2. 共通ナビゲーション / フッター

**ユーザーストーリー**
> すべてのユーザーとして、どのページからでも主要導線（占い師を探す・鑑定を探す・ブログ・ログイン）にアクセスしたい。

**受け入れ条件**
- [ ] AC-A2-1: ヘッダーに「占い師を探す」「鑑定を探す」「ブログ」「ログイン／マイページ」を配置する
- [ ] AC-A2-2: 占い師向け導線（「占い師として登録」）を区別して表示する（brighty の「コーチングをしたい方」相当）
- [ ] AC-A2-3: 未ログイン時は「ログイン」、ログイン時はロールに応じたマイページ導線を表示する
- [ ] AC-A2-4: フッターに利用規約・プライバシーポリシー・特定商取引法表記・FAQ・お問い合わせへのリンクを配置する
- [ ] AC-A2-5: モバイルでハンバーガーメニューに集約され、キーボード操作・スクリーンリーダーで操作可能

---

### エピックB: 占い師カタログ（既存機能踏襲＋刷新）

#### B-1. 占い師一覧

**ユーザーストーリー**
> 一般ユーザーとして、登録占い師の一覧をカード形式で閲覧し、気になる占い師を見つけたい。

**受け入れ条件**
- [ ] AC-B1-1: 占い師がカード（顔写真／アイコン、名前、得意占術タグ、対応相談形式、評価、簡潔な紹介）で一覧表示される
- [ ] AC-B1-2: ページネーションまたは無限スクロールで全件を閲覧できる
- [ ] AC-B1-3: 並び替え（おすすめ順／新着順／評価順）が選択できる
- [ ] AC-B1-4: 各カードから占い師プロフィール詳細へ遷移できる
- [ ] AC-B1-5: 占い師が0件のカテゴリ／検索条件では「該当する占い師がいません」を表示する（空状態）

#### B-2. 鑑定サービス一覧

**ユーザーストーリー**
> 一般ユーザーとして、提供されている鑑定サービスの一覧から、内容・形式・価格で比較検討したい。

**受け入れ条件**
- [ ] AC-B2-1: サービスがカード（タイトル、提供占い師、占術カテゴリ、相談形式、価格、所要時間）で一覧表示される
- [ ] AC-B2-2: 「占い師を探す」と「鑑定を探す」の2導線が独立して提供される（既存サイト踏襲）
- [ ] AC-B2-3: 各サービスカードからサービス詳細へ遷移できる

#### B-3. 検索・カテゴリ絞り込み

**ユーザーストーリー**
> 一般ユーザーとして、占術カテゴリ・相談形式・キーワードで占い師／サービスを絞り込み、自分に合う相手を効率的に見つけたい。

**受け入れ条件**
- [ ] AC-B3-1: キーワード検索（占い師名・サービス名・紹介文を対象）が可能
- [ ] AC-B3-2: 占術カテゴリ（15種）でフィルタできる。カテゴリは「6. 占術カテゴリ」に定義の15種
- [ ] AC-B3-3: 相談形式（電話／チャット／メール／Zoom／対面）でフィルタできる
- [ ] AC-B3-4: 価格帯・評価でフィルタできる
- [ ] AC-B3-5: 複数フィルタを AND 条件で組み合わせられる
- [ ] AC-B3-6: フィルタ条件はURLクエリに反映され、共有・リロードで再現できる
- [ ] AC-B3-7: フィルタ結果の件数がリアルタイムに表示される

#### B-4. 占い師プロフィール詳細

**ユーザーストーリー**
> 一般ユーザーとして、占い師の詳細プロフィールを見て、経歴・得意分野・提供サービス・口コミを確認し、相談するか判断したい。

**受け入れ条件**
- [ ] AC-B4-1: プロフィール（名前、写真、得意占術、経歴・自己紹介、対応相談形式、活動実績）を表示する
- [ ] AC-B4-2: その占い師が提供する鑑定サービス一覧を表示する
- [ ] AC-B4-3: その占い師が **著者として投稿したブログ記事一覧** を表示する（エピックCとの統合点）
- [ ] AC-B4-4: 「この占い師に相談する／予約をリクエストする」CTAを表示する
- [ ] AC-B4-5: 口コミ・評価が存在する場合は表示する（評価データが移行対象に含まれる場合）
- [ ] AC-B4-6: お気に入り登録ができる（ログイン時）

#### B-5. 鑑定サービス詳細

**ユーザーストーリー**
> 一般ユーザーとして、個別の鑑定サービスの内容・料金・所要時間・相談形式を確認し、予約リクエストを送りたい。

**受け入れ条件**
- [ ] AC-B5-1: サービス詳細（タイトル、説明、占術カテゴリ、相談形式、価格、所要時間、提供占い師）を表示する
- [ ] AC-B5-2: 提供占い師プロフィールへの導線を表示する
- [ ] AC-B5-3: 「このサービスを予約リクエストする」CTAを表示する
- [ ] AC-B5-4: 決済・実際の鑑定実施は **次フェーズ** であることをユーザーに明示する（予約リクエスト＝意思表示まで）

#### B-6. マッチング相談（brighty「アドバイザー相談」導線踏襲）

**ユーザーストーリー**
> 一般ユーザー（特に初心者）として、どの占い師・占術が自分に合うか分からないとき、相談内容を入力すると適した占い師の提案を受けたい。

**受け入れ条件**
- [ ] AC-B6-1: 「占い師選びを相談する」導線をトップ／一覧から提供する
- [ ] AC-B6-2: 相談フォームで悩みカテゴリ・希望相談形式・自由記述を入力できる
- [ ] AC-B6-3: 入力条件に基づき、マッチする占い師候補をルールベースで提示する（占術カテゴリ・相談形式・評価による条件マッチング。AI推薦は次フェーズ拡張余地として明記）
- [ ] AC-B6-4: 提示された候補から占い師プロフィール／予約リクエストへ遷移できる
- [ ] AC-B6-5: マッチング結果が0件の場合の代替提案（人気占い師など）を表示する

#### B-7. 予約リクエスト（相談リクエスト）

**ユーザーストーリー**
> 一般ユーザーとして、選んだ占い師・サービスに対して希望日時と相談内容を添えて予約リクエストを送り、占い師の応答を待ちたい。

**ユーザーストーリー（占い師）**
> 占い師として、届いた予約リクエストを確認し、承認・日程調整・辞退の応答をしたい。

**受け入れ条件**
- [ ] AC-B7-1: ログインユーザーが占い師／サービスに対し予約リクエスト（希望日時候補・相談形式・相談概要）を送信できる
- [ ] AC-B7-2: 送信後、ユーザーにリクエスト受付の確認が表示され、マイページで状態を確認できる
- [ ] AC-B7-3: 占い師は自分宛のリクエスト一覧を確認できる
- [ ] AC-B7-4: 占い師はリクエストに対し「承認／日程調整提案／辞退」の応答ができる
- [ ] AC-B7-5: リクエストの状態（pending / accepted / rescheduled / declined / cancelled）が双方で確認できる
- [ ] AC-B7-6: 状態変化時に通知（最低限アプリ内通知、メール通知は実装可能なら）を行う
- [ ] AC-B7-7: 実際の鑑定実施・決済は **次フェーズ**。MVPでは予約リクエストの成立（accepted）までをゴールとする

#### B-8. マイページ（一般ユーザー）

**受け入れ条件**
- [ ] AC-B8-1: お気に入り占い師一覧を表示する
- [ ] AC-B8-2: 送信した予約リクエストと状態を一覧表示する
- [ ] AC-B8-3: プロフィール（表示名・連絡先など最小限）を編集できる

#### B-9. 占い師ダッシュボード

**受け入れ条件**
- [ ] AC-B9-1: 自プロフィールを編集できる（得意占術、自己紹介、対応相談形式、写真）
- [ ] AC-B9-2: 自サービスの登録・編集・公開／非公開ができる
- [ ] AC-B9-3: 受信した予約リクエスト一覧と応答ができる（B-7と連携）
- [ ] AC-B9-4: 自分のブログ記事一覧・新規投稿への導線を持つ（エピックCと連携）

### エピックC: ブログ（新規 — 本体統合）

> 既存 `contents.uranai.cloud`（WordPress 想定）の役割を本体に統合する。占い師＝著者として記事を投稿できることが核心要件。

#### C-1. ブログ記事一覧・閲覧（公開）

**ユーザーストーリー**
> 訪問者として、占いコラム記事をカテゴリ・タグから探し、SEO的にも見つけやすい形で閲覧したい。

**受け入れ条件**
- [ ] AC-C1-1: 公開記事の一覧をカード形式（サムネイル、タイトル、抜粋、著者、公開日、カテゴリ）で表示する
- [ ] AC-C1-2: カテゴリ別・タグ別の記事一覧ページを提供する
- [ ] AC-C1-3: 記事詳細ページで本文・著者情報・関連記事・公開日を表示する
- [ ] AC-C1-4: 記事詳細から **著者（占い師）プロフィール** へ遷移できる（エピックBとの統合点）
- [ ] AC-C1-5: 記事内キーワード検索ができる
- [ ] AC-C1-6: ページネーションで全公開記事を閲覧できる
- [ ] AC-C1-7: 各記事に SEO メタ（title / description / OGP / JSON-LD Article）が出力される（詳細は `non-functional.md`）

#### C-2. 記事投稿・編集（運営・占い師ロール）

**ユーザーストーリー**
> 占い師として、自分の専門知識を記事として WYSIWYG エディタで執筆し、下書き保存・公開・予約投稿したい。

**ユーザーストーリー（運営）**
> 運営管理者として、全記事を作成・編集・公開承認・削除し、カテゴリ・タグを管理したい。

**受け入れ条件**
- [ ] AC-C2-1: 占い師・運営管理者ロールのみ記事投稿画面にアクセスできる（RBAC）
- [ ] AC-C2-2: WYSIWYG エディタで本文（見出し・装飾・リスト・リンク・画像挿入）を編集できる
- [ ] AC-C2-3: 記事にカテゴリ（1つ以上）とタグ（複数）を付与できる
- [ ] AC-C2-4: アイキャッチ画像をアップロードできる
- [ ] AC-C2-5: 記事を「下書き（draft）」として保存できる
- [ ] AC-C2-6: 記事を「公開（published）」にできる
- [ ] AC-C2-7: 公開日時を未来に指定し「予約投稿（scheduled）」できる。指定時刻に自動公開される
- [ ] AC-C2-8: 記事に SEO 設定（メタタイトル・メタディスクリプション・スラッグ・OGP画像）を任意で上書き設定できる
- [ ] AC-C2-9: 占い師が投稿した記事は **その占い師が著者** として記録され、プロフィールおよび記事に紐づく
- [ ] AC-C2-10: 占い師は自分の記事のみ編集・削除でき、他者の記事は編集できない（運営は全記事編集可）

#### C-3. ブログ管理（運営）

**受け入れ条件**
- [ ] AC-C3-1: 運営が全記事のステータス（draft / scheduled / published / archived）を一覧・変更できる
- [ ] AC-C3-2: 運営がカテゴリ・タグを作成・編集・削除できる
- [ ] AC-C3-3: 運営が記事の公開承認フロー（任意: 占い師投稿を承認後に公開）を運用できる設定を持つ
- [ ] AC-C3-4: 不適切記事の非公開（archive）操作ができる

#### C-4. ブログ⇔占い師カタログ統合

**受け入れ条件**
- [ ] AC-C4-1: 占い師プロフィールにその占い師の記事一覧が表示される（AC-B4-3 と対応）
- [ ] AC-C4-2: 記事詳細から著者占い師の「相談する」CTAへ動線がある（送客導線）
- [ ] AC-C4-3: 著者ページ（占い師＝著者のアーカイブ）が SEO 上有効な URL を持つ

## 4. データ移行要件

既存データ（占い師・ユーザー・サービス、および関連する評価・カテゴリ）を新スキーマへ移行する。

**受け入れ条件**
- [ ] AC-D-1: 占い師 101名分のプロフィールデータを移行する（名前、得意占術、紹介、相談形式、活動状態）
- [ ] AC-D-2: 鑑定サービス 59件分のデータを移行する（タイトル、カテゴリ、相談形式、価格、所要時間、提供占い師の関連付け）
- [ ] AC-D-3: 既存ユーザーアカウントを CC-Auth 連携前提で移行する（CC-Auth 側のアカウント突合方針は要確定 — 「7. 未確定事項」）
- [ ] AC-D-4: 占術カテゴリを15種（「6. 占術カテゴリ」）にマッピングする。既存カテゴリ表記の揺れは正規化する
- [ ] AC-D-5: 既存ブログ記事（`contents.uranai.cloud` / WordPress）の移行有無・方法を確定する（「7. 未確定事項」。MVPでは新規投稿基盤を必須とし、過去記事移行は移行元フォーマット確定後に判断）
- [ ] AC-D-6: 移行は冪等なマイグレーションスクリプトとして実装し、ドライラン・ロールバック手順を持つ

> **移行元フォーマット未確定**: 既存データの提供形式（DBダンプ / CSV / API / WordPress WXR 等）が未確定。詳細は「7. 未確定事項」。

## 7. 未確定事項・前提条件

実装着手前に確定が必要な事項。Phase 2/3 で解消する。

| # | 項目 | 内容 | 影響範囲 | 確定担当 |
|---|------|------|---------|---------|
| OPEN-1 | `.ccagi.yml` の `domain:` 未設定 | dev/prod サブドメインが未定義。本番URL・OGP絶対URL・sitemap に影響 | デプロイ・SEO | PO / 運営 |
| OPEN-2 | `.ccagi.yml` の `cc_auth:` 未設定 | CC-Auth の userPoolId / clientId / callback URL / localhost URL が未定義。認証・ロール連携に必須 | 認証全般 | PO / 基盤チーム |
| OPEN-3 | データ移行元フォーマット | 既存占い師101名・サービス59件・ユーザーの提供形式（DBダンプ/CSV/API/WXR）が未確定 | エピックB全般・移行スクリプト | 運営 / 旧システム管理者 |
| OPEN-4 | 既存ユーザーとCC-Authの突合 | 既存ユーザーアカウントをCC-Authへどう紐付けるか（メール突合／再登録／インポート） | 認証・移行 | PO / 基盤チーム |
| OPEN-5 | 既存ブログ記事の移行可否 | `contents.uranai.cloud` 過去記事を移行するか、新規のみか。移行する場合 WordPress WXR 等のエクスポート形式 | エピックC | 運営 |
| OPEN-6 | 評価・口コミデータの有無 | 既存に占い師評価データが存在し移行対象か | AC-B4-5 / AC-B1-3 | 運営 |
| OPEN-7 | 予約成立後の連絡手段 | MVPで予約accepted後の双方連絡をアプリ内のみとするか、メール／外部手段を含むか | AC-B7-6 | PO |
| OPEN-8 | ブログ公開承認フローの要否 | 占い師投稿を運営承認後に公開するか、即時公開か | AC-C3-3 | 運営 |

> **GLOBAL GUARDRAILS 準拠**: OPEN-1 / OPEN-2 は `.ccagi.yml` に未記載のため、本書では「要設定」と明記する。AWS デプロイ／CC-Auth 連携着手前に確定すること。
