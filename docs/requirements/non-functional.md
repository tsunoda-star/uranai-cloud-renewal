## 1. パフォーマンス要件

### 1.1 Lighthouse スコア目標

| カテゴリ | 目標値 | 必達 |
|---------|--------|------|
| Performance | 90+ | ✅ |
| Accessibility | 100 | ✅ |
| Best Practices | 100 | ✅ |
| SEO | 100 | ✅ |

計測対象ページ: トップ、占い師一覧、占い師プロフィール、ブログ記事詳細（モバイル・デスクトップ両方）。

### 1.2 Core Web Vitals

| 指標 | 目標値 | 説明 |
|------|--------|------|
| LCP（Largest Contentful Paint） | < 2.5s | 主要コンテンツ描画 |
| INP（Interaction to Next Paint） | < 200ms | インタラクション応答 |
| CLS（Cumulative Layout Shift） | < 0.1 | レイアウトシフト |
| FCP（First Contentful Paint） | < 1.8s | 初回描画 |
| TTFB（Time to First Byte） | < 0.8s | サーバー応答 |

### 1.3 実装方針（Next.js 15 App Router）

- [ ] PERF-1: Server Components を既定とし、Client Components は必要箇所（フォーム、フィルタUI、WYSIWYGエディタ）に限定する
- [ ] PERF-2: 占い師一覧・ブログ一覧は適切なキャッシュ戦略（ISR / `revalidate`）を用い、初回表示を高速化する
- [ ] PERF-3: 画像は `next/image` で最適化（WebP/AVIF、適切な `sizes`、遅延読み込み）。占い師写真・アイキャッチに適用
- [ ] PERF-4: フォントは `next/font` でセルフホスト・サブセット化し、レイアウトシフトを防止（`font-display: swap`）
- [ ] PERF-5: 検索・フィルタはサーバーサイドで実行し、クエリパラメータベースでキャッシュ可能にする
- [ ] PERF-6: バンドルサイズを監視し、ルートごとのクライアントJSを最小化する
- [ ] PERF-7: Prisma クエリは N+1 を回避し、必要なリレーションのみ select する

## 2. SEO 要件

ブログ本体統合（エピックC）とカタログの検索流入が事業価値に直結するため、SEO は必達要件とする。

### 2.1 メタデータ

- [ ] SEO-1: 全ページで Next.js Metadata API により `title` / `description` を動的生成する
- [ ] SEO-2: 占い師プロフィール・サービス詳細・ブログ記事は、エンティティ固有の title / description を出力する
- [ ] SEO-3: ブログ記事は著者が任意でメタタイトル・メタディスクリプション・スラッグを上書きできる（AC-C2-8 と対応）
- [ ] SEO-4: 重複・薄いコンテンツに対し `canonical` URL を適切に設定する

### 2.2 OGP / Twitter Card

- [ ] SEO-5: 全主要ページで OGP（`og:title`, `og:description`, `og:image`, `og:type`, `og:url`）を出力する
- [ ] SEO-6: ブログ記事はアイキャッチを `og:image` に、占い師プロフィールは写真を `og:image` に使用する
- [ ] SEO-7: Twitter Card（`summary_large_image`）を出力する
- [ ] SEO-8: OGP の絶対URLは環境変数（本番ドメイン）から生成する。ドメインは `.ccagi.yml` 設定値を使用（**未設定 = 要設定**、OPEN-1 参照）

### 2.3 構造化データ（JSON-LD）

- [ ] SEO-9: ブログ記事に `Article`（または `BlogPosting`）構造化データを出力（headline / author / datePublished / image）
- [ ] SEO-10: 占い師プロフィールに `Person` または `ProfilePage` 構造化データを出力
- [ ] SEO-11: サイト全体に `Organization` / `WebSite`（SearchAction 含む）を出力
- [ ] SEO-12: パンくずに `BreadcrumbList` を出力

### 2.4 サイトマップ / クローラビリティ

- [ ] SEO-13: `sitemap.xml` を動的生成し、公開占い師・公開サービス・公開ブログ記事・カテゴリページを含める（下書き・予約投稿前の記事は除外）
- [ ] SEO-14: `robots.txt` を提供し、管理画面・マイページ・APIをクロール対象外にする
- [ ] SEO-15: URL は人間可読なスラッグ構造（例: `/advisors/[slug]`, `/blog/[slug]`, `/blog/category/[category]`）とする
- [ ] SEO-16: ブログ記事URLは公開後に不変とし、変更時は 301 リダイレクトを設定する
- [ ] SEO-17: 旧サイト（`www.uranai.cloud` / `contents.uranai.cloud`）からの主要URLは可能な範囲で 301 リダイレクトマッピングを用意する（移行元URL確定後 — OPEN-3/OPEN-5）

## 3. アクセシビリティ要件（WCAG 2.1 AA）

- [ ] A11Y-1: WCAG 2.1 レベル AA に準拠する
- [ ] A11Y-2: テキストのコントラスト比 4.5:1 以上、大きい文字・UIコンポーネントは 3:1 以上を満たす
- [ ] A11Y-3: すべての画像に適切な `alt` を付与する（装飾画像は空 alt）
- [ ] A11Y-4: すべての操作要素がキーボードのみで操作可能。フォーカス順序が論理的、フォーカスリングが視認可能
- [ ] A11Y-5: フォーム要素に `label` を関連付け、エラーはテキストで明示しスクリーンリーダーに伝わる
- [ ] A11Y-6: ランドマーク（header / nav / main / footer）と適切な見出し階層（h1 は1ページ1つ）を用いる
- [ ] A11Y-7: 動的UI（フィルタ結果、通知、モーダル）に適切な ARIA 属性・`aria-live` を付与する
- [ ] A11Y-8: タッチターゲットは 44×44px 以上（AC-A1-6 と対応）
- [ ] A11Y-9: `prefers-reduced-motion` を尊重し、アニメーションを抑制できる
- [ ] A11Y-10: 色のみに依存して情報を伝えない（状態表示はアイコン・テキスト併用）

## 4. セキュリティ要件

- [ ] SEC-1: 認証・認可は CC-Auth に委譲。アプリは CC-Auth 発行トークンを検証し RBAC を実施する（OPEN-2: cc_auth 設定要）
- [ ] SEC-2: 3ロール（一般／占い師／運営）の権限境界をサーバーサイドで強制する。クライアント側の制御のみに依存しない
- [ ] SEC-3: 記事編集・サービス編集は所有者または運営のみ（AC-C2-10）。サーバーアクション／API で所有権を必ず検証する
- [ ] SEC-4: ユーザー入力（検索・フォーム・記事本文）は適切にバリデーション・サニタイズする
- [ ] SEC-5: WYSIWYG 由来のHTMLは XSS 対策のためサニタイズ（許可タグのホワイトリスト）を行う
- [ ] SEC-6: SQL は Prisma 経由でパラメータ化し、生クエリは原則使用しない
- [ ] SEC-7: 機密情報（CC-Auth クライアントシークレット、DB接続情報、APIキー）は環境変数で管理し、リポジトリにハードコードしない
- [ ] SEC-8: CSRF / クリックジャッキング対策（適切なセキュリティヘッダ: CSP, X-Frame-Options, HSTS 等）を設定する
- [ ] SEC-9: アップロード画像（プロフィール写真・アイキャッチ）はファイル種別・サイズを検証し、安全なストレージに保存する
- [ ] SEC-10: レート制限を予約リクエスト送信・マッチング相談・ログイン試行に適用する
- [ ] SEC-11: 個人情報（相談内容・連絡先）は最小限の保持とし、アクセスログを管理者操作に対して記録する
- [ ] SEC-12: 依存パッケージの脆弱性スキャンを CI に組み込む

> CCAGI 規約に従い、ソースコード変更時は IronDome（API ルートセキュリティ）スキャンを Phase 4 以降で適用する。

## 5. レスポンシブ要件（モバイルファースト）

- [ ] RWD-1: モバイルファーストで設計・実装する（占い相談はモバイル利用が主と想定）
- [ ] RWD-2: 以下のブレークポイントをサポートする

| 名称 | 幅 | 主対象 |
|------|-----|--------|
| sm | 640px | 大型スマートフォン |
| md | 768px | タブレット縦 |
| lg | 1024px | タブレット横・小型ノート |
| xl | 1280px | デスクトップ |
| 2xl | 1536px | 大型デスクトップ |

- [ ] RWD-3: 375px（標準スマートフォン）でレイアウト崩れ・横スクロールが発生しない
- [ ] RWD-4: ナビゲーションはモバイルでハンバーガー、デスクトップで水平メニューに切り替わる
- [ ] RWD-5: カードグリッド（占い師・サービス・ブログ）はビューポートに応じて列数を可変にする（1→2→3→4列）
- [ ] RWD-6: 画像は各ブレークポイントで適切な解像度を配信する（`next/image` の `sizes`）
- [ ] RWD-7: タッチ操作とマウス操作の双方で操作性を担保する

## 6. 可用性・信頼性

- [ ] REL-1: データ移行は冪等・ロールバック可能なマイグレーションとする（AC-D-6 と対応）
- [ ] REL-2: 予約リクエストの状態遷移はトランザクション整合性を保つ
- [ ] REL-3: 予約投稿（scheduled）の自動公開は確実に実行される仕組み（cron / スケジュールジョブ）を持つ（AC-C2-7 と対応）
- [ ] REL-4: エラー時はユーザーに分かりやすいフォールバック（エラーページ・空状態）を表示する
- [ ] REL-5: 主要操作（予約リクエスト、記事公開）の失敗はログに記録し、運営が追跡できる
